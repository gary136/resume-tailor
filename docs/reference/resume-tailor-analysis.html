Resume Tailor — case 1 builds the master, case 2 spends it

Analysis of the two scenarios you described: interactive resume tailoring (case 1) and an automated job-search-and-apply pipeline (case 2). They are not separate apps — case 1 is the prerequisite phase that produces the master template and your preference record; case 2 is the automated phase that consumes them.

The core insight: case 1 is the foundry, case 2 is the factory

Both phases run the same essential operation — take a resume, take a "direction" (your stated emphasis in case 1, a job description in case 2), and produce a tailored result. But the relationship between them is sequential, not parallel:

  • Phase 1 (case 1) — interactive sessions generate the master: a polished resume, a fact inventory extracted from it, and a preference record (your voice, what edits you accept/reject). Direction comes from you; every change is confirmed.
  • Phase 2 (case 2) — the pipeline runs on those outputs: it can't score fit without the master's fact inventory, and can't tailor in your voice without the preference record. Direction comes from fetched job descriptions; approval is automatic or gated (see Q3).

So case 1 isn't just the first feature to build — its outputs are the input contract of case 2. The pipeline is only ever as good as the master and preferences that phase 1 produced, which is also why returning to case 1 sessions periodically (new achievements, updated skills) keeps phase 2 sharp.

Phase 1 (case 1) — interactive tailoring, and forging the master

The flow you described is a confirm-then-edit loop. Its end product is more than one tailored resume: the confirmed result becomes the master template, and the session's accept/reject decisions become the preference record. Two design choices make or break it:

  • Propose direction before touching text. The agent should first present a short plan — "you want to stress X, so I'd promote these two bullets, compress the early roles, and rewrite the summary around X" — and get a yes/no before rewriting. Cheaper to correct a plan than a draft.
  • Show edits as diffs, section by section. Never hand back a whole new document to proofread. Accepting/rejecting per-section changes is faster and builds the preference record ("keeps metrics in bullets", "hates buzzwords") that case 2 later relies on.

Phase 2 (case 2) — the automated pipeline

Runs only once phase 1 has produced a confirmed master and fact inventory. Your description maps cleanly to a pipeline, with one refinement worth calling out: when a new job description arrives, score it against the whole resume library, not just the master. You already hinted at this ("store as another template for possible future reuse") — the pipeline should close that loop, so the tenth backend-engineer posting reuses the backend variant instead of generating an eleventh.

Note the third branch your description didn't have: fundamentally unqualified. "Resume doesn't fit" has two very different causes — wrong presentation (fixable by tailoring) and wrong person (a director-level role for an IC, a required license you don't hold). Tailoring must only fix the first; the second gets skipped with a logged reason. Without this branch the system will happily fabricate qualifications, which brings us to risks.

The hard problems, honestly ranked

ProblemDifficultyReality check
Auto-apply hardest This is the make-or-break piece. Major boards (LinkedIn, Indeed) prohibit bot applications in their ToS and defend with CAPTCHAs and login walls. What is realistic: applying directly on company career pages built on a handful of ATS platforms (Greenhouse, Lever, Ashby) whose forms are predictable, via browser automation. Everything else falls to the manual queue — expect the queue to dominate early on, which is fine: "manual queue with the right resume pre-attached" is already most of the value.
Job sourcing medium Same ToS issue on scraping. Prefer legitimate feeds: job-board APIs (Adzuna, JSearch, USAJobs), ATS public postings APIs (Greenhouse/Lever boards are openly queryable), RSS. Which sources matter depends on your target market — see Q2.
Truthfulness guardrail medium The tailoring engine must be constrained to reorder, rephrase, emphasize, and trim — never invent experience, inflate titles, or add skills. Concretely: extract a fact inventory from the master once (roles, dates, skills, achievements); every variant is validated against it — any claim not traceable to the inventory is rejected. This is what makes the "fundamentally unqualified" branch enforceable.
Fit scoring tractable A good LLM rubric: separate hard requirements (met / not met) from soft alignment (0–100). Hard-requirement misses → skip branch; low soft score with hard requirements met → tailor branch. Thresholds will need tuning against your judgment for the first few dozen jobs — case 1 sessions help calibrate.
Variant sprawl easy, if early One variant per job description → 200 near-duplicates. Instead, key variants to a job family ("senior backend — fintech"), reuse when a new JD matches an existing family, and only allow small per-application overlays (e.g. swapping a keyword) on top.

One position to take before building: fully unattended submission is where this gets risky — a bug that mass-sends a bad resume is unrecoverable, and an application is an outward-facing act made in your name. My recommendation: the pipeline prepares everything, and submission itself waits behind a one-tap approval (batched, e.g. "12 ready — review tonight"). Graduate to full auto only for ATS platforms + job families with a proven track record.

What gets stored

Five stores, all simple: the master + fact inventory (truth), variants keyed by job family, the preference record case 1 accumulates, the application log (essential — it also prevents double-applying to the same company), and the manual queue. SQLite plus markdown resume files covers all of it; no infrastructure needed for personal use.

Parallel development tracks

The tailoring engine and the apply engine can be built in parallel — but not because they're "different features." They're parallelizable because they never call each other: each one only reads and writes shared data records. The apply engine doesn't care how a resume was tailored; it takes a resume file + a job record and tries to submit. That means it can be developed against a dummy resume before the tailoring engine exists at all.

Notes on the tracks:

  • Track C should start as a spike, not a build. The apply engine is the highest-risk component and its findings reshape scope: a two-day probe answering "which ATS forms can we actually fill?" tells you whether stage 4 is worth building at all — better to learn that in week 1 than after the rest is done. Until it lands, everything routes to the manual queue, which works fine.
  • Track B's fit evaluator doesn't need to wait for Track A — develop and tune it against a hand-written fact inventory of your real resume, swap in the parser's output later.
  • What must stay sequential lives inside Track A: auto-tailoring (A3) genuinely depends on interactive tailoring (A2), because the preference record and the confirmed master are its inputs — this is the phase-1-as-prerequisite dependency, and no amount of parallelism removes it.
  • Parallel only pays if there are parallel workers. Solo, the stage order below is still the right walk. With AI agents doing the building, the three tracks are a textbook case for parallel agents in separate worktrees — independent code, integration only at the contract layer.

Suggested delivery order

Orthogonal to the tracks above: tracks say what can be built concurrently; stages say in what order value is delivered and used. Each stage is a working tool on its own.

  1. Phase 1: resume library + interactive tailoring. Not just the first feature — the prerequisite: it generates the master, the fact inventory, and the preference record that every later stage runs on.
  2. Job sourcing + fit scoring → manual queue. The pipeline minus auto-apply. Already delivers most of case 2's value: wake up to a ranked list of jobs, each with the right resume attached.
  3. Auto-tailoring for "no fit" jobs, with the truthfulness validator.
  4. Auto-apply, one ATS platform at a time, behind the approval gate.

If stage 4 turns out to be more fight than value (the Track C spike will tell you early), stages 1–3 still stand.

Decisions (2026-07-18): POC first, product if it works

What the POC → product split means for the build

  • The canonical resume format is the bridge. Your product answer to Q5 — "transform any upload to a same-format output to the engine" — is exactly the data-contract layer from the tracks section. Design that canonical schema on day 1 of the POC, and the POC's parser (PDF → canonical) becomes the first of the product's importers rather than throwaway code.
  • Build the core as a library, the CLI as a shell. If the tailoring engine, fit evaluator, and apply engine live in a package the CLI merely calls, the product's web app wraps the same package — the POC→product jump is then a UI + accounts project, not a rewrite.
  • Phase 1 begins with authoring, not importing. Since no existing resume is usable as a starting point, the POC's first interactive session is an intake interview — the agent gathers your facts (roles, dates, achievements, skills) directly from you, builds the fact inventory first, and writes the master from it. This is actually the cleaner path: the fact inventory becomes the source of truth from day 1 instead of being reverse-engineered from a PDF, and the parser/importer drops out of the POC's critical path entirely (it returns at product stage for other users' uploads).
  • US-only keeps POC sourcing simple — Greenhouse/Lever/Ashby public boards + an aggregator API. Structure connectors behind one interface so TW sources (104, CakeResume) slot in at product stage without touching the pipeline.
  • "Fully unattended if possible" at product stage matches the graduated-trust model in the callout above: unattended is earned per ATS platform + job family with a proven track record, never the default. For a product, this maturing also carries the liability question — submissions go out in the user's name, so the trust ladder should be per-user opt-in.
  • Product-stage flags to revisit later (not POC concerns): per-user credential handling for application accounts, job-board ToS exposure at commercial scale, and resume-data privacy (it's PII by definition).
