The one true serial chain is petrol: intake → tailoring loop → auto-tailoring. Stage 3 cannot start early because its inputs — the confirmed master and your accumulated accept/reject record — only exist once you've used 1b and 1c. Everything else decouples.

The three tracks

Track A — Resume

stages 1b → 1c → 3
  • Intake interview → fact inventory → author master (fresh, never imported)
  • Interactive loop: propose → confirm → per-section diffs → accept/reject
  • Auto-tailoring reuses variants by job family

Bottleneck: you. 1b and 1c need you live — facts and taste are user-only information.

Track B — Jobs

stage 2
  • Greenhouse / Lever / Ashby boards + aggregator behind one connector interface
  • Fit evaluator: hard-requirements gate, then soft score → fits / tailor / skip
  • Manual queue + application log (double-apply blocked at the DB)

Agent-buildable now against a fixture inventory; swaps to your real facts the day 1b lands.

Track C — Apply

spike → stage 4
  • Spike: can Playwright fill a Greenhouse/Lever form with a dummy resume?
  • If yes: one ATS adapter at a time, dev against dummy data
  • Every submission sits behind batched approval — no exceptions

Fully agent-buildable; the apply engine never cares how a resume was tailored.

Suggested parallel schedule

Three waves. Within a wave, rows run concurrently — one track is yours, the others are delegable. Waves are dependency boundaries, not calendar weeks.

Rules that make the parallelism safe

frozen contracts
Tracks meet only at the schemas in docs/reference/data-contracts.md. Changing one now requires a dated note there — that's what keeps three concurrent streams from drifting apart.
fixtures first
Tracks B and C develop against dummy fact inventories and dummy resumes. Real data flows in when track A produces it; no track waits on another to write code.
no fabrication
Every resume bullet must carry <!-- facts: … --> refs into the inventory; the validator rejects anything else. Auto-tailoring inherits this for free.
approval gate
Nothing is submitted anywhere without your batch approval. In the store, submitted is unreachable except through approved, and only your action sets that.
one shot per job
UNIQUE(job_id) in the application log — double-applying is physically impossible, so parallel queue-fillers can't collide.
